# Архитектура

## Назначение

`moving-checklist-bot` — Telegram-бот для общего чеклиста переезда двух пользователей: владельца и партнёра.

## Основные модули

`bot.py` загружает конфиг, открывает SQLite, создаёт Gemini-парсер, подключает роутеры aiogram и запускает polling.

`app/config.py` читает `.env`, валидирует `BOT_TOKEN`, требует `GEMINI_API_KEY`, задаёт модель Gemini и путь `data/bot.db`.

`app/database/storage.py` содержит SQLite-схему и async-методы доступа. Таблицы:

- `users` — владелец и партнёр.
- `invitations` — одноразовые коды входа.
- `checklist_items` — общий список вещей.
- `action_logs` и `action_log_items` — история последних действий для отката.

`app/handlers/` содержит роутеры:

- `start.py` — `/start`, первый владелец, вход по приглашению и отказ без доступа.
- `invitations.py` — `/invite` для владельца.
- `checklist.py` — кнопки списков, свободный текст, откат.
- `errors.py` — общий обработчик ошибок.

`app/services/gemini.py` изолирует вызовы `google-genai` и разбор JSON-ответа. Тесты используют мок-парсер и не делают реальные запросы.

`app/services/formatting.py` форматирует списки и подтверждения для Telegram.

## Поток обработки текста

1. Пользователь отправляет свободный текст.
2. Хендлер проверяет доступ.
3. Бот берёт активные вещи из базы и передаёт их в Gemini вместе с текстом.
4. Gemini возвращает строгий JSON-массив действий `add` и `mark_done`.
5. База транзакционно добавляет новые вещи, отмечает выполненные и пишет action log.
6. Пользователь получает короткое подтверждение.
7. Кнопка «↩️ Отменить последнее» откатывает последний action log этого пользователя.

## Ограничения

- Все пользователи работают с одним общим списком.
- Приглашение одноразовое.
- Нет миграционного инструмента: схема создаётся через `CREATE TABLE IF NOT EXISTS`.
